# IntuiTerm: The Ultimate Technical & Architectural Guide

This document serves as a comprehensive reference for the IntuiTerm project. It is designed to empower a developer (or a student in a job interview) to explain every design decision, folder structure, and technical trade-off made during development.

---

## 1. Project Philosophy & Core Objective
IntuiTerm is a **Cloud-Native Terminal Emulator** for Windows. It bridges the gap between the raw power of a Linux-like terminal and the high-fidelity UI expectations of modern web applications. The goal is to provide a "Terminal-as-a-Platform" where AI assistance, file management, and system telemetry coexist in a single, high-performance interface.

---

## 2. Detailed Folder Structure: The "Why" Behind the Layout

The project follows a **Decoupled Monorepo** structure. This means the frontend and backend live in the same repository for ease of development but are built as independent, loosely coupled services.

### `backend/` - The Engine Room
- **`app/api/`**: Contains the route definitions. 
    *   *Why?* We separate routing from business logic. If we want to move from FastAPI to another framework later, we only change the routing layer.
- **`app/services/`**: The core business logic (Terminal management, AI translation).
    *   *Why?* Services are "stateless" workers. This modularity allows us to test the AI service without even starting the terminal.
- **`app/core/`**: Configuration and global settings (API URLs, system constants).
- **`bin/`**: Portable binaries (BusyBox, Nano).
    *   *Why?* By bundling these, we ensure the project is "Portable." It doesn't rely on the user having Linux installed; it brings its own environment.

### `frontend/` - The User Experience
- **`src/components/`**: Atomic and Molecular UI components.
    *   *Why?* We follow a "Single Responsibility Principle." `TerminalPane` only handles rendering; `FileExplorer` only handles the tree view.
- **`src/hooks/`**: Custom React logic (e.g., `useTerminalSessions`).
    *   *Why?* Hooks extract complex state logic out of the UI, making the components cleaner and easier to read.
- **`src/assets/`**: Static images and icons.
- **`tailwind.config.js` & `index.css`**: The design system.

### `linux_env/` - The Virtualized Layer
- Contains the Alpine Linux filesystem and storage images. This provides the "Linux feel" on a Windows host.

## 3. Modularization: Strategy & Trade-offs

### Why Modularize? (The Advantages)
1.  **Scalability**: We can add a "Network Monitor" component just by creating a new file in `components/` and a new route in `api/`.
2.  **Maintainability**: If the AI Assistant breaks, we know exactly where to look (`services/ai.py`) without digging through terminal logic.
3.  **Collaborative-Ready**: Multiple developers can work on different services (one on the File Explorer, one on the Terminal PTY) without causing code conflicts.
4.  **Testability**: Each module can be unit-tested in isolation.

### The Disadvantages (The "Cost" of Clean Code)
1.  **Abstraction Overhead**: Sometimes it takes 3 files to do something that could be done in 1. This requires more mental mapping of the project.
2.  **Configuration Complexity**: Wiring together multiple modules (Frontend <-> API <-> Service <-> Binary) requires robust configuration management (`core/config.py`).

## 4. Coding Conventions & Standards

### Backend (Python)
- **Asynchronous Patterns**: We use `async/await` extensively. This is "Non-blocking," meaning the server can handle 100 users typing in terminals simultaneously without slowing down.
- **Type Hinting**: All functions use Python 3.10+ type hints (e.g., `def run(cmd: str) -> None`). This makes the code self-documenting.
- **PEP 8 Compliance**: Following standard Python style guides for naming and spacing.

### Frontend (TypeScript/React)
- **Functional Components**: We use modern React Hooks (`useState`, `useEffect`) instead of old Class components.
- **Strict Typing**: TypeScript interfaces are used for all data (e.g., `TerminalSession` interface). This prevents "Undefined is not a function" errors.
- **CSS-First Design**: We use Tailwind for layout but rely on CSS Variables (Tokens) for the theme. This allows for "One-Click" theme switching.

## 5. Technical Deep-Dive (Interview Cheat Sheet)

### Why FastAPI instead of Flask?
*   **FastAPI** is built on Starlette and Pydantic, making it one of the fastest Python frameworks available. Its native support for WebSockets and Asynchronous programming makes it the superior choice for a real-time terminal.

### Why Vite instead of Create-React-App (Webpack)?
*   **Vite** uses "Native ESM" to serve files during development. It doesn't bundle the whole app every time you save; it only updates the changed file. This reduces "Save-to-Refresh" time from 5 seconds to 50 milliseconds.

### WebSockets vs. HTTP Polling
*   **HTTP Polling** is like asking "Are we there yet?" every second. It's wasteful.
*   **WebSockets** are a single, open "pipe." When a user presses 'A' in the terminal, it's sent instantly. When the server has output, it's pushed instantly. This is "Real-time" vs. "Near-real-time."

### PTY (Pseudo-Terminal) vs. Shell Execution
*   Running `subprocess.run('ls')` is a one-shot command.
*   A **PTY** is an interactive session. It allows things like `vim`, `nano`, or `top` to run because it emulates a real hardware terminal. Our `services/terminal.py` manages these PTYs.

## 6. Security Architecture
- **Environment Isolation**: We use `.env` files (managed via `os.getenv`) for sensitive API keys.
- **Input Sanitization**: Terminal inputs are handled through the PTY layer, which prevents simple shell injection attacks against the host Python process.
- **CORS (Cross-Origin Resource Sharing)**: The backend is configured to only accept requests from the IntuiTerm frontend, preventing unauthorized external access.
